\chapter{Introduction}
\section{Background}
Libraries manage multiple categories of information simultaneously: bibliographic records, copy-level inventory, member records, lending transactions, reservations, financial dues, administrative content, audit logs, and user engagement data. Traditional single-database implementations often store all of these concerns in one relational schema, but that approach can make some workloads inefficient. Transaction processing, document-heavy content, analytics, and recommendation graphs have different access patterns and modelling needs.

This project addresses that challenge by implementing a \textbf{Library Management System} using a polyglot data architecture. Instead of forcing every module into one storage model, the application assigns each workload to the database technology that best matches its characteristics.

\section{Problem Statement}
The problem addressed by the project is the lack of an integrated institution-scale library application that simultaneously provides:
\begin{itemize}[leftmargin=1.5em]
    \item accurate relational transaction handling for borrowing and reservations,
    \item traceable administrative and financial records,
    \item flexible content and activity storage,
    \item graph-based recommendation capabilities,
    \item role-aware access for members, staff, and administrators,
    \item and a unified web interface for both public and administrative use.
\end{itemize}

\section{Objectives}
The major objectives of the project are:
\begin{enumerate}[leftmargin=1.5em]
    \item To build a centralized digital platform for managing books, members, staff, and circulation transactions.
    \item To automate critical library workflows such as checkout, return, fine processing, reservation creation, and dashboard reporting.
    \item To demonstrate the practical application of polyglot persistence in a realistic database systems project.
    \item To maintain auditability, data quality, and role-based control across administrative operations.
    \item To support enhanced member experience through reviews, wishlist, ebooks, and personalized recommendations.
    \item To provide a maintainable architecture with clear separation between frontend, backend, and database layers.
\end{enumerate}

\section{Scope of the Project}
The implemented system covers:
\begin{itemize}[leftmargin=1.5em]
    \item public catalog browsing and book detail pages,
    \item member authentication and dashboard access,
    \item librarian and administrator operations,
    \item book catalog management,
    \item member management,
    \item circulation and return-desk workflows,
    \item reservation handling,
    \item analytics and reporting,
    \item review, rating, and moderation features,
    \item ebook administration and member digital reading support,
    \item content management and announcement publishing,
    \item financial ledger and audit trail management,
    \item graph-based recommendations.
\end{itemize}

The current scope does not include inter-library loan exchange, external procurement integration, barcode hardware integration, SMS gateways, or a dedicated mobile application.

\section{Non-Functional Requirements}
\subsection{Performance}
The system requires acceptable response time for catalog browsing, dashboard loading, and administrative screens. To support this:
\begin{itemize}[leftmargin=1.5em]
    \item PostgreSQL indexes are created for high-frequency book and transaction access patterns.
    \item MongoDB aggregation pipelines are used for analytics-heavy workloads.
    \item Neo4j handles graph traversal for recommendations instead of deeply nested relational joins.
    \item In-memory caching is used for selected dashboard and report responses.
\end{itemize}

\subsection{Scalability}
Scalability is improved through workload separation:
\begin{itemize}[leftmargin=1.5em]
    \item transactional data remains in PostgreSQL,
    \item audit and ledger reporting move to MySQL,
    \item flexible log and document collections are placed in MongoDB,
    \item relationship-intensive recommendations are offloaded to Neo4j.
\end{itemize}

\subsection{Reliability}
Reliability is addressed through:
\begin{itemize}[leftmargin=1.5em]
    \item health-check endpoints,
    \item connection pooling,
    \item graceful shutdown,
    \item asynchronous logging queue with retry control,
    \item non-blocking secondary-store sync for optional services.
\end{itemize}

\subsection{Security}
Security requirements include:
\begin{itemize}[leftmargin=1.5em]
    \item JWT-based authentication,
    \item role-based authorization for member, staff, and admin actions,
    \item PostgreSQL row-level security support in the schema and migration scripts,
    \item financial and audit traceability,
    \item constrained file upload types and file size limits.
\end{itemize}

\section{Feasibility}
\subsection{Technical Feasibility}
The project is technically feasible because it uses mature technologies with strong ecosystem support: React for UI composition, Express for API management, Prisma for relational ORM access, and dedicated database engines for specialized workloads.

\subsection{Operational Feasibility}
The system is operationally feasible for a college-scale or departmental library setting because it provides admin panels, health status visibility, modular routes, and environment-driven configuration.

\subsection{Economic Feasibility}
The architecture is based primarily on open-source frameworks and drivers. Managed cloud services can be used, but the implementation also supports local development, which reduces development cost.

\chapter{System Architecture}
\section{Architectural Overview}
The system follows a multi-tier architecture:
\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Presentation Layer}: React and Vite frontend.
    \item \textbf{Application Layer}: Express backend with domain-specific route modules.
    \item \textbf{Data Access Layer}: Prisma, database drivers, analytics modules, and data-store adapters.
    \item \textbf{Persistence Layer}: PostgreSQL, MySQL, MongoDB, and Neo4j.
\end{enumerate}

\section{High-Level Request Flow}
\begin{enumerate}[leftmargin=1.5em]
    \item A user interacts with the frontend through public, member, or admin pages.
    \item The frontend calls REST endpoints exposed by the Express API.
    \item Middleware performs parsing, authentication, authorization, and error handling.
    \item Route modules execute business logic using Prisma or dedicated database drivers.
    \item Responses are returned to the frontend, while selected events are logged asynchronously and some data is synchronized to secondary stores such as Neo4j or MySQL.
\end{enumerate}

\section{Dynamic Data Store Adapter}
The backend includes a \texttt{DataStoreAdapter} abstraction that can route selected workloads across MongoDB, MySQL, or PostgreSQL using configuration. It is used for activity logs, audit trail, and analytics-related storage settings. This design reduces hardcoded dependencies and improves configurability for experimentation and deployment.

\section{Caching and Asynchronous Logging}
The backend contains:
\begin{itemize}[leftmargin=1.5em]
    \item an in-memory cache for frequently requested report and dashboard fragments,
    \item prefix-based invalidation after state-changing operations,
    \item an asynchronous logging queue with bounded size, retry support, and configurable concurrency.
\end{itemize}

\section{PostgreSQL Indexing Strategy}
The system employs indexing for both correctness and performance:
\begin{itemize}[leftmargin=1.5em]
    \item standard B-tree indexes on identifiers, status, dates, categories, and relations,
    \item partial indexes for active rows and frequently filtered states,
    \item GIN trigram indexes on title, author, ISBN, and member names for text search,
    \item composite indexes on transaction status and due date for overdue checks.
\end{itemize}

Database-side logic is a major part of the system design.

\subsection{Triggers}
The migration file includes database triggers for:
\begin{itemize}[leftmargin=1.5em]
    \item automatic \texttt{updated\_at} maintenance,
    \item inventory count adjustment during checkout and return,
    \item fine calculation on return events.
\end{itemize}

\section{MySQL for Audit and Financial Records}
MySQL stores administrative trace and financial ledger information. The principal tables are:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{audit\_trail}: field-level or event-level changes related to books and circulation,
    \item \texttt{member\_audit\_log}: member record and PII access changes,
    \item \texttt{financial\_records}: fine, payment, refund, and waiver ledger entries.
\end{itemize}

This separation is important because financial and audit records are append-oriented and reporting-heavy. Keeping them apart from the main circulation tables avoids overloading the transactional source of truth.

\section{MongoDB for Flexible Documents and Analytics}
MongoDB is used in several distinct roles:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{activity\_logs} for user and system activity events,
    \item \texttt{cms\_pages} for dynamic page-section content,
    \item \texttt{announcements} for notice publishing,
    \item \texttt{reviews} and \texttt{review\_reactions} for review features,
    \item \texttt{analytics} for job history and analytics snapshots.
\end{itemize}

MongoDB indexes are explicitly created for activity timestamps, action filtering, page and section uniqueness, reviews by book and member, and reaction uniqueness.

\section{Neo4j for Recommendation Graph}
Neo4j models:
\begin{itemize}[leftmargin=1.5em]
    \item nodes: \texttt{Book}, \texttt{Author}, \texttt{Category}, \texttt{Member}, \texttt{Tag}, \texttt{Ebook},
    \item relationships: \texttt{WRITTEN\_BY}, \texttt{BELONGS\_TO}, \texttt{BORROWED}, \texttt{INTERESTED\_IN}, \texttt{HAS\_TAG}, \texttt{LIKED\_REVIEW\_OF}, \texttt{WISHLISTED}, \texttt{BASED\_ON}, \texttt{UPLOADED}.
\end{itemize}

Neo4j is treated as a recommendation-oriented read model rather than the transactional source of truth. Synchronization is performed after key PostgreSQL events.

\section{Data Synchronization Model}
Selected application events update secondary databases:
\begin{itemize}[leftmargin=1.5em]
    \item new or updated books are synchronized to Neo4j,
    \item member signup is synchronized to Neo4j,
    \item checkout and return operations create graph edges or update relationship state,
    \item payment and fine events are logged to MySQL,
    \item activity events are inserted into MongoDB asynchronously,
    \item ebook uploads are synchronized to Neo4j.
\end{itemize}

These synchronizations are intentionally non-blocking where possible, so that optional analytics or recommendation services do not prevent completion of the primary user workflow.

\chapter{Module Design and Implementation}
\section{Authentication and Authorization}
Authentication is JWT-based. The middleware supports two token sources:
\begin{itemize}[leftmargin=1.5em]
    \item Supabase-style JWTs using \texttt{SUPABASE\_JWT\_SECRET},
    \item legacy application JWTs using \texttt{SECRET\_KEY}.
\end{itemize}

Authorization is role-sensitive. All users are resolved from the \texttt{members} table, and the middleware exposes:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{requireMember} for active member access,
    \item \texttt{requireStaff} for staff, librarian, or admin access,
    \item \texttt{requireAdmin} for administrator-only actions.
\end{itemize}

\section{Catalog Management Module}
The catalog module provides:
\begin{itemize}[leftmargin=1.5em]
    \item public book listing and filtering,
    \item per-book detail retrieval,
    \item ISBN lookup,
    \item admin create, update, and delete operations,
    \item cover metadata, availability counts, and reference-only restrictions.
\end{itemize}

On catalog changes, the backend can log audit data and synchronize core catalog attributes to Neo4j.

\section{Member Management Module}
Member management supports:
\begin{itemize}[leftmargin=1.5em]
    \item member registration,
    \item login,
    \item current-user lookup,
    \item member listing and profile updates,
    \item role and account-state aware access handling.
\end{itemize}

\section{Circulation Module}
Circulation is one of the central modules of the project. The workflow includes:
\begin{enumerate}[leftmargin=1.5em]
    \item verifying that the selected book exists, is active, is not reference-only, and has available copies,
    \item verifying that the selected member exists, is active, has no blocking dues, and does not already hold the same book,
    \item creating a checkout transaction,
    \item updating visit metadata,
    \item recording activity logs and audit records,
    \item synchronizing borrow information to Neo4j,
    \item returning books, calculating fines, updating copy state, and recording financial entries if fines are produced.
\end{enumerate}

The circulation desk UI also records checkout and return condition states for books, which is an operationally relevant library feature.

\section{Reservation Module}
The reservation module allows members to place and cancel reservations while staff can inspect all reservations. Core logic includes:
\begin{itemize}[leftmargin=1.5em]
    \item active-member validation,
    \item outstanding fine checks,
    \item prevention of duplicate pending reservations,
    \item automatic queue position assignment,
    \item expiry date computation from site settings,
    \item cancellation note handling and audit logging.
\end{itemize}

\section{Collections and Content Management Module}
The system supports editorial and administrative content using a combination of PostgreSQL and MongoDB:
\begin{itemize}[leftmargin=1.5em]
    \item curated collections and pinned collections are stored in PostgreSQL,
    \item site settings are stored in PostgreSQL key-value form,
    \item CMS page sections are stored in MongoDB,
    \item staff board posts are stored in PostgreSQL,
    \item announcements are stored in MongoDB,
    \item news content is managed separately through backend routes.
\end{itemize}

\section{Digital Library Module}
There are two digital content paths:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{Administrative ebooks}: uploaded by administrators, stored in the \texttt{ebooks} table, and optionally exposed publicly.
    \item \textbf{User library uploads}: uploaded by members, stored in \texttt{user\_uploaded\_books}, and readable through streaming routes.
\end{itemize}

The system also tracks reading progress in PostgreSQL using \texttt{reading\_progress}, making it possible to resume digital reading sessions.

\section{Reviews, Ratings, and Engagement Module}
The review subsystem is hybrid by design:
\begin{itemize}[leftmargin=1.5em]
    \item structured numeric ratings are stored in PostgreSQL (\texttt{book\_ratings}),
    \item review documents, comments, replies, and reactions are stored in MongoDB,
    \item review-like events are synchronized to Neo4j for recommendation enrichment,
    \item GIF search and trending support is provided through a Giphy integration route.
\end{itemize}

This is a strong example of polyglot modelling inside a single feature domain.

\section{Analytics and Reporting Module}
The reporting layer is divided by workload:
\begin{itemize}[leftmargin=1.5em]
    \item MongoDB: activity dashboards and behavioural trends,
    \item MySQL: financial and audit summaries,
    \item PostgreSQL: collection analytics and circulation reports,
    \item combined overview endpoint: aggregates multiple sources in one response.
\end{itemize}

The admin reports page periodically refreshes key metrics, demonstrating live operational reporting.

\section{Recommendation and Wishlist Module}
The recommendation subsystem provides:
\begin{itemize}[leftmargin=1.5em]
    \item popular books,
    \item related books,
    \item personalized recommendations,
    \item trending books for a period,
    \item also-borrowed suggestions,
    \item member reading-profile insights,
    \item wishlist and wishlist-based recommendations.
\end{itemize}

\section{Financial and Audit Module}
Financial operations combine PostgreSQL and MySQL:
\begin{itemize}[leftmargin=1.5em]
    \item PostgreSQL functions update member fines and generate dashboard values,
    \item MySQL records immutable ledger entries for payments, fines, waivers, and refunds,
    \item MySQL also stores audit trail records linked to book and circulation events.
\end{itemize}

\chapter{Query Design and Database Logic}
\section{Representative PostgreSQL Queries}
The following query patterns are central to the system.

\subsection{Basic Query: Active Loans}
\begin{lstlisting}[language=SQL]
SELECT *
FROM transactions
WHERE status IN ('checked_out', 'overdue')
ORDER BY checkout_date DESC;
\end{lstlisting}
This query supports the circulation desk and active-loan displays.

\subsection{Advanced Query: Popular Books by Circulation}
\begin{lstlisting}[language=SQL]
SELECT b.id AS "bookId",
       b.title,
       b.author,
       COUNT(DISTINCT t.id) AS "checkoutCount",
       COUNT(DISTINCT r.id) AS "reservationCount"
FROM books b
LEFT JOIN transactions t ON t.book_id = b.id
LEFT JOIN reservations r ON r.book_id = b.id
WHERE b.is_active = true
GROUP BY b.id, b.title, b.author
HAVING COUNT(DISTINCT t.id) > 0
ORDER BY "checkoutCount" DESC, "reservationCount" DESC;
\end{lstlisting}
This query is used for collection analytics and demand analysis.

\subsection{Advanced Query: Availability Forecast}
\begin{lstlisting}[language=SQL]
SELECT b.id AS "bookId",
       b.title,
       COUNT(DISTINCT r.id) AS "waitlistCount",
       MIN(t.due_date) AS "earliestDueDate"
FROM books b
LEFT JOIN reservations r
    ON r.book_id = b.id AND r.status = 'pending'
LEFT JOIN transactions t
    ON t.book_id = b.id
   AND t.status = 'checked_out'
   AND t.return_date IS NULL
WHERE b.available_copies = 0
GROUP BY b.id, b.title
HAVING COUNT(DISTINCT r.id) > 0;
\end{lstlisting}
This query predicts pressure points in library demand.

\subsection{Inventory Update Trigger}
This trigger adjusts available copy counts automatically when a transaction is inserted or updated.

\begin{lstlisting}[language=SQL]
CREATE OR REPLACE FUNCTION public.fn_update_available_copies()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' AND NEW.status = 'checked_out' THEN
    UPDATE public.books
    SET available_copies = GREATEST(0, available_copies - 1)
    WHERE id = NEW.book_id;

  ELSIF TG_OP = 'UPDATE'
    AND OLD.status = 'checked_out'
    AND NEW.status = 'returned' THEN
    UPDATE public.books
    SET available_copies = LEAST(total_copies, available_copies + 1)
    WHERE id = NEW.book_id;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_update_copies ON public.transactions;
CREATE TRIGGER trg_update_copies
  AFTER INSERT OR UPDATE ON public.transactions
  FOR EACH ROW
  EXECUTE FUNCTION public.fn_update_available_copies();
\end{lstlisting}

\subsection{Fine Calculation Trigger}
This trigger calculates overdue fines when a checked-out book is returned late.

\begin{lstlisting}[language=SQL]
CREATE OR REPLACE FUNCTION public.fn_calculate_fine()
RETURNS TRIGGER AS $$
DECLARE
  v_days_overdue int;
  v_daily_rate numeric := 0.50;
  v_max_cap numeric := 20.00;
  v_fine numeric := 0;
BEGIN
  IF NEW.status = 'returned'
     AND NEW.return_date IS NOT NULL
     AND OLD.status = 'checked_out' THEN

    v_days_overdue := GREATEST(
      0,
      EXTRACT(DAY FROM (NEW.return_date::timestamptz - NEW.due_date::timestamptz))::int
    );

    IF v_days_overdue > 0 THEN
      v_fine := LEAST(v_days_overdue * v_daily_rate, v_max_cap);
      NEW.fine_amount := v_fine;

      UPDATE public.members
      SET fines = fines + v_fine
      WHERE id = NEW.member_id;
    END IF;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_calculate_fine ON public.transactions;
CREATE TRIGGER trg_calculate_fine
  BEFORE UPDATE ON public.transactions
  FOR EACH ROW
  EXECUTE FUNCTION public.fn_calculate_fine();
\end{lstlisting}

\subsection{Payment Processing Function}
This database-side function is called by the financial API to reduce outstanding fines and mark eligible fine transactions as paid.

\begin{lstlisting}[language=SQL]
CREATE OR REPLACE FUNCTION public.fn_process_payment(
  p_member_id int,
  p_amount numeric,
  p_processed_by text DEFAULT 'system'
)
RETURNS TABLE(remaining_balance numeric) AS $$
BEGIN
  UPDATE public.members
  SET fines = GREATEST(0, fines - p_amount)
  WHERE id = p_member_id;

  UPDATE public.transactions
  SET fine_paid = true
  WHERE id IN (
    SELECT id FROM public.transactions
    WHERE member_id = p_member_id
      AND fine_amount > 0
      AND fine_paid = false
    ORDER BY checkout_date ASC
    LIMIT 50
  )
  AND fine_amount <= p_amount;

  RETURN QUERY
  SELECT fines FROM public.members WHERE id = p_member_id;
END;
$$ LANGUAGE plpgsql;
\end{lstlisting}

\begin{lstlisting}[language=SQL]
SELECT member_id,
       SUM(CASE WHEN transaction_type = 'fine' THEN amount ELSE 0 END) AS total_fined,
       SUM(CASE WHEN transaction_type = 'payment' THEN amount ELSE 0 END) AS total_paid,
       SUM(CASE WHEN transaction_type = 'fine' THEN amount ELSE 0 END)
       - SUM(CASE WHEN transaction_type IN ('payment','waiver') THEN amount ELSE 0 END)
         AS outstanding
FROM financial_records
GROUP BY member_id
HAVING outstanding > 0
ORDER BY outstanding DESC;
\end{lstlisting}

\section{Business Logic Placement}
An important design decision in this project is deliberate distribution of logic:
\begin{itemize}[leftmargin=1.5em]

\chapter{Conclusion}
The \textbf{Library Management System} delivers a complete and well-structured solution for modern library operations. It supports catalog management, member administration, circulation, reservations, digital reading, reviews, analytics, financial tracking, and recommendation features within one connected platform. PostgreSQL ensures reliability for core transactions, MySQL provides clear audit and financial visibility, MongoDB handles flexible content and analytics efficiently, and Neo4j adds relationship-driven intelligence that improves discovery and personalization.

The overall result is a system that is practical, scalable, and modular. It reduces manual work, improves traceability, supports better decision-making through analytics, and creates a stronger experience for both library staff and members. By combining multiple database models in a purposeful way, the project achieves far more than a basic library application and shows how a well-designed data architecture can directly improve system capability, maintainability, and user experience.
